# ADR-019 — Bridge Observability

**Status:** Proposed
**Date:** 2026-04-18
**Deciders:** Helena (IT Manager)
**Related:** ADR-017 (Tier 2 Bridge Surface), ADR-018 (Bridge Security Boundary)

---

## Context

ADR-017 defines the `renderShortcodes` entry point and its `BridgeInput`/`BridgeOutput`
contract. ADR-018 §6 mandates baseline logging per invocation (info, warn, error) and
requires a `trace_id` compatible with the server observability system. Both ADRs explicitly
defer detailed tracing to this ADR.

As of Sprint 2, NodePress has no OpenTelemetry SDK wired into the Node runtime. Structured
JSON logs via `console.log` are the operational baseline. This ADR defines:

1. What data is captured per bridge invocation.
2. Where it is emitted (JS wrapper, not PHP side).
3. The minimum log format for the PoC and the interface shape for OTel adoption in Sprint 3+.
4. What is explicitly excluded for privacy reasons.
5. Alert conditions (warn vs. error).

---

## Decision

Every invocation of `renderShortcodes` MUST emit a structured log entry before returning.
The log entry is emitted by the JS bridge wrapper — not by PHP code. PHP does not own
observability; it only feeds raw signals (duration, warnings array) that the JS layer
normalises and emits.

The implementation for Sprint 2 is `console.log(JSON.stringify(span))` using the interface
defined in §Span Shape. The interface is OTel-ready: Sprint 3+ can swap the emit call
for an OTel `Span` without changing the shape or the collection logic.

---

## What Is Traced

Each bridge invocation emits **one span** with the following fields. All fields are
mandatory unless marked optional.

### Span Shape

```ts
// packages/bridge/src/observability.ts

export interface BridgeSpan {
  /**
   * Span type identifier. Always "bridge.render".
   * Allows log aggregators to filter bridge spans from other structured logs.
   */
  readonly event: "bridge.render";

  /**
   * Correlation ID propagated from BridgeMeta.traceId.
   * If the caller did not provide a traceId, the bridge generates a UUID v4.
   * Enables correlation with REST layer traces and upstream request logs.
   */
  readonly trace_id: string;

  /**
   * Per-invocation ID from BridgeMeta.invocationId (also drives open_basedir
   * per ADR-018 §2). UUID v4, generated by the bridge if absent.
   */
  readonly invocation_id: string;

  /**
   * Identifier of the plugin that owns the shortcodes being rendered.
   * Maps to BRIDGE_PLUGIN_ID or the plugin's declared id. Never null.
   */
  readonly plugin_id: string;

  /**
   * Primary shortcode tag being expanded in this invocation.
   * For multi-shortcode content, the first matched tag. "unknown" if no
   * shortcode was detected before PHP ran (content passes through anyway).
   */
  readonly shortcode_tag: string;

  /**
   * Wall-clock duration of the full renderShortcodes call in milliseconds,
   * including input validation, PHP execution, and output parsing.
   * Measured in JS with performance.now() before and after the await.
   */
  readonly duration_ms: number;

  /**
   * Byte length of BridgeInput.postContent (UTF-8 encoded).
   * Used to correlate latency with content size.
   */
  readonly input_size_bytes: number;

  /**
   * Byte length of BridgeOutput.html (UTF-8 encoded).
   * Zero if the render failed and passthrough was applied.
   */
  readonly output_size_bytes: number;

  /**
   * BridgeError.code if the render failed, null if successful.
   * Values: BRIDGE_TIMEOUT | BRIDGE_OOM | BRIDGE_FATAL |
   *         BRIDGE_INIT_FAILED | BRIDGE_INPUT_REJECTED | null
   */
  readonly error_code: string | null;

  /**
   * Number of non-fatal warnings emitted by this invocation.
   * Corresponds to BridgeOutput.warnings.length.
   * Does NOT include warning content — content may carry PII (see §What Is Not Traced).
   */
  readonly warnings_count: number;

  /**
   * ISO-8601 timestamp of when the span was emitted (end of invocation).
   * Used for time-series aggregation and log ordering.
   */
  readonly timestamp: string;
}
```

### Emit Point

The span is emitted in `renderShortcodes` **before the function returns**, after
`BridgeOutput` is assembled. Pseudocode:

```ts
export async function renderShortcodes(
  input: BridgeInput,
): Promise<BridgeOutput> {
  const t0 = performance.now();
  const invocationId = input.meta?.invocationId ?? crypto.randomUUID();
  const traceId = input.meta?.traceId ?? crypto.randomUUID();

  let output: BridgeOutput;
  try {
    output = await executeWithTimeout(input, invocationId);
  } catch (err) {
    output = buildPassthrough(input, err);
  }

  const span: BridgeSpan = {
    event: "bridge.render",
    trace_id: traceId,
    invocation_id: invocationId,
    plugin_id: BRIDGE_PLUGIN_ID,
    shortcode_tag: detectPrimaryTag(input.postContent),
    duration_ms: Math.round(performance.now() - t0),
    input_size_bytes: Buffer.byteLength(input.postContent, "utf8"),
    output_size_bytes: Buffer.byteLength(output.html, "utf8"),
    error_code: output.error?.code ?? null,
    warnings_count: output.warnings.length,
    timestamp: new Date().toISOString(),
  };

  emitSpan(span); // Sprint 2: console.log(JSON.stringify(span))
  return output;
}
```

### Emit Function (Sprint 2 implementation)

```ts
// packages/bridge/src/observability.ts

export function emitSpan(span: BridgeSpan): void {
  console.log(JSON.stringify(span));
}
```

The function signature is stable. Sprint 3+ replaces the body with an OTel
`tracer.startSpan` / `span.end()` call. No other code changes are required.

## What Is NOT Traced

The following are explicitly excluded. Logging any of these constitutes a security/privacy
violation and blocks PR review.

| Excluded field                                              | Reason                                                                                                    |
| ----------------------------------------------------------- | --------------------------------------------------------------------------------------------------------- |
| `BridgeInput.postContent` (full text)                       | May contain PII (user-generated content, drafts, private posts).                                          |
| `BridgeOutput.html` (full text)                             | Derived from post content — same PII risk.                                                                |
| Shortcode attribute values                                  | Attributes may contain user-supplied data (email addresses, names, API keys passed as config).            |
| `BridgeInput.context.pluginConfig` values                   | Plugin config values may contain API keys or user-identifiable settings.                                  |
| `BridgeInput.context.candidatePosts[*].excerpt` or `.title` | User-generated content.                                                                                   |
| PHP stack traces                                            | Internal server paths. Captured by the bridge logger (ADR-018 §6) but never included in the span.         |
| `warnings[]` content                                        | Warning strings may reflect attribute values that triggered a stub. Only `warnings_count` is in the span. |

## Alert Conditions

The following conditions determine the log level of the emitted span. Log level is
determined in `emitSpan` based on `BridgeSpan` fields.

| Condition            | Log level | Trigger                                                                    |
| -------------------- | --------- | -------------------------------------------------------------------------- |
| Successful render    | `info`    | `error_code === null && duration_ms < 2000`                                |
| Slow render          | `warn`    | `error_code === null && duration_ms >= 2000` (approaching 3s hard timeout) |
| Warnings present     | `warn`    | `error_code === null && warnings_count > 0`                                |
| Input rejected       | `warn`    | `error_code === "BRIDGE_INPUT_REJECTED"`                                   |
| PHP OOM              | `error`   | `error_code === "BRIDGE_OOM"`                                              |
| PHP timeout          | `error`   | `error_code === "BRIDGE_TIMEOUT"`                                          |
| PHP fatal            | `error`   | `error_code === "BRIDGE_FATAL"`                                            |
| Runtime boot failure | `error`   | `error_code === "BRIDGE_INIT_FAILED"`                                      |

For `warn` and `error` spans, the emitter adds a `level` field to the JSON object so
log aggregators can filter without parsing `error_code`:

```ts
export function emitSpan(span: BridgeSpan): void {
  const level = resolveLevel(span);
  console.log(JSON.stringify({ level, ...span }));
}

function resolveLevel(span: BridgeSpan): "info" | "warn" | "error" {
  if (
    span.error_code === "BRIDGE_OOM" ||
    span.error_code === "BRIDGE_TIMEOUT" ||
    span.error_code === "BRIDGE_FATAL" ||
    span.error_code === "BRIDGE_INIT_FAILED"
  )
    return "error";
  if (
    span.error_code !== null ||
    span.warnings_count > 0 ||
    span.duration_ms >= 2000
  )
    return "warn";
  return "info";
}
```

## Tool and Implementation Scope

**Sprint 2 (this ADR):**

- Implementation: `console.log(JSON.stringify(...))` with `level` field.
- No OTel SDK. No external collector. No dashboards.
- Output goes to stdout. In CI, stdout is captured by the runner log. In staging,
  stdout is collected by the process manager (PM2 or Docker compose log driver).

**Sprint 3+ (not defined here, requires new ADR or amendment):**

- Replace `emitSpan` body with `@opentelemetry/api` Tracer calls.
- Wire `trace_id` and `invocation_id` into OTel span attributes.
- Add metrics: `bridge.invocation.duration` histogram, `bridge.error.count` counter.
- Dashboard: latency p50/p95/p99 per `plugin_id` and `shortcode_tag`.

The `BridgeSpan` interface and `emitSpan` signature are designed to make this migration
additive (no breaking changes to callers).

## Consequences

### What this locks in

1. `renderShortcodes` MUST emit a `BridgeSpan` on every invocation — success, failure,
   and passthrough. No silent invocations.

2. The `BridgeSpan` interface is frozen for Sprint 2. Adding fields is allowed (additive).
   Removing or renaming fields requires amending this ADR.

3. The `emitSpan` function is the single emit point. Direct `console.log` calls with bridge
   data outside `emitSpan` are prohibited — all observability goes through the defined
   interface.

4. Privacy constraints in §What Is Not Traced are non-negotiable. Violations are a security
   incident, not a code style issue. Enforcement: automated test verifying span shape does
   not include prohibited fields.

5. ADR-018 §6 logging (stub warn, timeout error, etc.) is handled by the bridge internals
   and feeds into `warnings_count` and `error_code`. ADR-018 does not contradict this ADR
   — they operate at different layers (internal bridge events vs. per-invocation span).

### Positive

- One span per invocation gives full per-call visibility: latency, size, errors, warnings.
- OTel-ready interface means zero rework when SDK is added.
- Privacy-by-design: field exclusion list is explicit and enforced by tests, not convention.
- `level` field in emitted JSON enables grep-level triage without log parsing tools.

### Negative

- `shortcode_tag` detection from raw content is heuristic (first `[tag]` match). Multi-shortcode
  content reports only the primary tag. Full per-shortcode granularity requires a separate span
  per tag — deferred to Sprint 3+.
- `console.log` is not structured-log-safe if the process runs with a logger that intercepts
  stdout (e.g., Pino, Winston). Sprint 2 assumption: stdout goes to runner/Docker logs directly.
  Integration with app logger is Sprint 3 work.
- No sampling in Sprint 2. Every invocation is logged. At high RPS this generates volume.
  Sampling strategy is Sprint 3+ concern.
